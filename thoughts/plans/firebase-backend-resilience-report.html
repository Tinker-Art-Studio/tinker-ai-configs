Highest risk: if the data is lost or corrupted today, recovery depends on manual backups and scattered in-app safeguards — not on a single, owned, tested recovery plan. The stakes are concrete: Classbook has already lost teacher content twice, Tinker Ticker is the source of truth for payroll, Summer Camp can overwrite or delete shared data in bulk, and Training Hub holds staff compliance records that cannot be recreated.

In plain terms, the goal is to: turn on Firebase's built-in automatic backups (PITR plus scheduled backups); keep a second backup copy somewhere normal app or agent credentials cannot delete; confirm that deleted photos and files can actually be recovered; write down and practice how to restore; limit who — and which AI agents — can change production; and add automated tests — written test-first (TDD) and as plain-language behavior specs (BDD); see Adopting TDD and BDD — that assume an agent will eventually attempt an unsafe change unless something stops it.

2. Where the resilience work stands today — and what to fix first

Plain summary: some good safety work has already started, mostly in Classbook. But the backbone of any recovery plan — a real, automatic, off-machine backup — does not exist yet. Almost everything that protects this data today depends on a person remembering to run something by hand, or on safety code that has not been finished. This section lays out what is actually done, what is still open, and the order to tackle it. It is the prioritized view; the sections that follow give the full target state and the step-by-step roadmap.

The single most important fact: there is no managed, automatic backup of the Firebase data. The only backup tool in the code is a script an admin has to paste into a browser by hand, roughly once a week. The 30-minute automatic backup that rescued Classbook during the May data loss ran on one person's Mac as a local scheduled job — and it is not even present on this machine. If that one Mac is lost, wiped, or simply not run, there is currently no way to recover. Confirming what exists in the Firebase console and replacing this with a managed backup is the highest priority, ahead of everything else below.

5. Recommended backend continuity target state

One backup, but per-app recovery. All four apps share a single Firebase project and a single Firestore database, so Point-in-Time Recovery (PITR) and scheduled backups protect every app at the same time — there is no way to have them on for one app and off for another; they apply to the whole database. The catch is on the recovery side: a managed restore rebuilds the entire database into a new one, so you must never restore the whole database over production to fix a single app — that would overwrite the other three. To recover one app on its own, the plan is:
  1. Restore into an isolated sandbox — bring the full snapshot, or a PITR timestamp, back into a separate database/project (never production).
  2. Surgically copy back only the affected app's collections from that sandbox into production, leaving the other apps untouched.
  3. Keep per-app collection-scoped exports — schedule exports limited to each app's own collections to its own backup path, so a single app's data can also be re-imported independently without a full restore.
For very recent loss, PITR can additionally read specific documents as of a past time and write just those back. Net result: one database-wide safety net, plus the ability to recover Classbook, Summer Camp, Tinker Ticker, or Training Hub individually.

Firestore data protection

  1. Enable Point-in-Time Recovery (PITR) on the default Firestore database for short-window accidental writes/deletes.
  2. Enable scheduled Firestore backups with at least daily cadence and the maximum retention that fits budget/compliance needs. Firebase docs currently describe daily/weekly backup schedules with restore into a new database, and PITR up to a 7-day window.
  3. Keep independent export copies in a locked Cloud Storage bucket or separate Google Cloud project/account so a compromised app deploy path cannot erase both production and backups.
  4. Define RPO/RTO by data class: payroll/timeclock and active camp content should have RPO ≤ 1 hour during business operations; lower-critical seed/catalog data can tolerate daily RPO.
  5. Design for per-app recovery. All four apps share one database, so PITR/scheduled backups are database-wide (one setting protects everyone). To recover a single app without disturbing the others, do not restore the whole database over production; instead restore the snapshot or PITR timestamp into an isolated database, then copy back only that app's collections. Also run per-app collection-scoped exports (each app's collections to its own backup path) so one app can be re-imported independently. See the callout above.

Cloud Storage protection

  1. Verify Cloud Storage soft delete retention on the Firebase bucket and document exact retention days.
  2. Enable object versioning or retention/lifecycle rules for high-value paths: curriculum/, summerCamps/, projectDetailPhotos/, and training/.
  3. Run a separate scheduled object inventory/export for Storage metadata and object manifests, because Firestore exports only preserve references/URLs, not binary content.
  4. Prohibit delete-before-upload replacement flows; upload new object, verify URL/readability, then remove old object after a grace period.

Security and agent-defense controls

  1. Use least-privilege IAM, MFA, and a small human-only production owner group. Agents should not hold Owner/Editor or broad Firebase Admin rights.
  2. Disable or tightly govern service-account key creation. Prefer Workload Identity / short-lived credentials for CI jobs.
  3. Require PR review for studio-hub/firestore.rules, deploy scripts, Netlify env vars, backup scripts, and Firebase config changes.
  4. Introduce an explicit emergency break-glass account owned by humans, not agents, with logged use and credential rotation after incidents.

Restore readiness

  1. Quarterly restore drills into a non-production Firestore database/project.
  2. Per-app restore runbooks that identify collections, expected document counts, owner sign-off, and validation checks.
  3. Read-only daily integrity reports: document counts, critical field non-empty counts, recent destructive writes, last backup age.
  4. Incident checklist: freeze deploys, revoke suspect credentials, preserve audit logs, restore to staging first, then surgically patch production.

Root causes and contributing factors

  • Payload semantics misunderstood: Firestore merge does not protect fields included with empty strings.
  • Over-broad writes: a one-lesson edit originally triggered a save path capable of touching many lesson documents.
  • Generated empty state mixed with persisted authored state: generated lesson slots and teacher-authored content shared objects, making stale/blank client state dangerous.
  • Silent load failure: before Stage 1A, failed reads could leave the UI apparently usable with blank content.
  • Patch incompleteness: the first fix protected bulk saves but missed the single-lesson path until the second wipe forced a full save-path audit.

Retrospective recommendations

  1. Finish the remaining safety-plan stages: photo upload-before-delete, Plan Complete targeted writes, admin move/swap read-back, non-summer stripping, post-save verification, backup health visibility, and content-count anomaly detection.
  2. Promote the current local backup approach into managed Firestore backups/PITR plus an independent export with visible status; keep ≤30-minute effective RPO during active curriculum/camp editing until managed controls prove equivalent.
  3. Make “no full-object writes for partial edits” a repository invariant enforced by tests and PR review.
  4. Require any future agent touching Firebase write paths to run the Classbook data-safety Playwright suite and include the pass/fail output in review.

Repository hygiene and cross-repo consistency

Good code that never leaves a laptop protects nothing. This report's headline failure — the backup that rescued Classbook living only on one person's Mac — is the same risk that applies to agent-written code: a fix that is never committed, pushed, and deployed is not a fix. With roughly two dozen Tinker repos, safeguards only hold if every repo carries the same guidance and every agent plans and executes the same way. These are three checks to make routine.

1. Commit-and-push completeness check (end of every task)

  1. Working tree is clean. git status shows no modified, staged-but-uncommitted, or untracked files left behind. Generated artifacts are either committed or explicitly ignored — not silently abandoned.
  2. All work is committed in reviewable units, following the commit protocol above (risky backend changes in their own commits with evidence).
  3. The branch is pushed to the remote and a PR is opened; nothing important exists only in the local clone.
  4. "Done" includes the deploy path. For changes that must reach production (rules, data-safety fixes), the task is not complete until the approved deploy has run or is explicitly queued for human approval, with the deployed commit recorded.
  5. The agent states completion explicitly: branch name, PR link, clean-tree confirmation, and what remains for a human (review, deploy, console step).

2. Per-repo guidance (CLAUDE.md / AGENTS.md) in every repo

  1. Every repo contains an up-to-date guidance file stating the shared Firebase-safety invariants: no full-object writes for partial edits; strip empty fields; await critical writes; snapshot before bulk import/delete; never run bare firebase deploy; the authoritative rules source is studio-hub; how to run that repo's data-safety/rules tests.
  2. The shared invariants live in one canonical place; each repo's file points to it rather than drifting into per-repo variants.
  3. A periodic cross-repo audit confirms the guidance file is present and current in all repos, and flags any repo that is missing it or has fallen behind the canonical version.

3. Shared skills for consistent planning and execution

  1. Build reusable skills/commands once and reuse them across all repos so planning and execution do not depend on which agent or repo is involved — for example: a data-safety pre-flight, a backup-before-bulk gate, a pre-deploy checklist, and a standard planning workflow.
  2. Require these skills for any Firebase-risk work, and record in the PR/work log which skill was used.
  3. Keep the skills versioned centrally so an improvement (a new safety check) propagates to every repo at once, instead of being re-implemented unevenly.

9. Adopting test-first development (TDD) and behavior specs (BDD)

This belongs in a continuity report for one reason: if AI agents write most of the code, the automated tests are the guardrail that survives a careless agent. Every data-loss incident in this report came from a write that did the wrong thing silently. A test that asserts the right behavior — and fails loudly when the behavior is wrong — is the cheapest, most durable defense available. TDD (test-driven development) and BDD (behavior-driven development) are two complementary disciplines for producing those tests.

This is not a from-scratch ask. These projects already have the seeds of it: Classbook ships a Playwright data-safety suite where Tests 5 and 6 are written and intentionally failing to mark safeguards that haven't been built yet (that is TDD), and studio-hub/rules.test.js contains ~29 emulator-backed rules checks that read like behavior specifications (that is BDD in spirit). The recommendation is to make this the standard for every Firebase-risk change, and to fill the gaps where tests are missing or only check mocks.

What each one means here

  • TDD — write the test first, watch it fail, then make it pass. The cycle is red → green → refactor: write a test that encodes a safety rule (for example, "a save containing an empty field must not erase saved content"), run it and confirm it fails (red), write the smallest code that makes it pass (green), then tidy up. The discipline guarantees the test actually catches the bug, because you watched it fail before the fix existed. Classbook's failing Tests 5 and 6 are exactly this: finishing those safeguards means turning red tests green.
  • BDD — describe the behavior in plain language before building it. Each rule is written as a Given / When / Then scenario that a non-technical owner can read and confirm: it doubles as the specification and the test. Tools such as Cucumber/Gherkin formalize the format, but the real value is a shared, jargon-free description of what "correct" means, agreed before code is written and kept in sync with the test that enforces it.

Example behavior specs (BDD scenarios)

These read the same to an engineer and to a business owner. Each one becomes an automated test that runs against the Firebase emulator or a real test project, and each maps directly to a risk named earlier in this report.

10. Development process and practice

The earlier sections defined the safety rules: how agents should write, commit, and test risky changes (agent-defense, commit history as a record, repo hygiene, test-first development). This section is the operating model that makes those rules hold across all of Tinker's roughly two dozen repositories at once — and ends with a single setup prompt to stand it up.

Why this is a continuity issue, not a developer-preference issue. The most important thing surfaced in the June 18 working session was not a specific bug — it was a pattern: "it said it fixed it, it wasn't fixed; it said it fixed it, it wasn't fixed." Classbook's second wipe happened because the first fix was never verified across every save path. When AI agents do nearly all the building, the process around the agent is the continuity control. A reliable process is what turns "the agent says it's safe" into "we can prove it's safe."

Framing (consistent with the April 2026 platform audit): this is phased hardening of the development operating system, not a rebuild. Each recommendation below states the practice and the continuity payoff. None of it has to be adopted all at once — the setup prompt installs the scaffolding, and the practices are adopted one app at a time.

2. Keep the skills in their own repo, and never let them drift

By default Claude writes skills into the local machine (~/.claude or ~/.agents). That is the same single-machine failure mode as the Classbook backup that rescued production but lived only on one Mac — if the laptop is lost, the operating system that keeps the apps safe is lost with it.

  1. One dedicated skills repository (the ai-configs pattern), pushed to GitHub, is the source of truth for all skills, commands, and shared agent guidance.
  2. It installs into the agent directory (for example, the repo's skills/ folder is symlinked into ~/.claude/skills) so edits in the repo take effect immediately.
  3. Every change is committed and pushed — automatically. Because agents reliably drift back to "edit locally, forget to commit," add a real mechanism: a Claude Stop hook or a short "save skills" routine that, after any edit under the skills directory, commits and pushes. Treat a skill change that is not pushed as not done, exactly like the commit-and-push completeness check for app code.

3. Plan first — with the HTML plan reviewer in the loop

Christie's stated blocker for the Classbook safety work was trust: "I don't even know how to trust its safety plan." The answer is to make the plan a reviewable artifact and harden it before any risky code runs — which is exactly how this report was produced.

  • Install the plan reviewer locally (it runs on the laptop; nothing is shared externally). It serves plans and documents to a browser where you highlight any element, leave a comment, and an agent picks the comment up and revises the document.
  • Route non-trivial work through it. For anything touching a Firebase write path, Claude publishes a plan; Christie reviews and comments in the browser; Claude revises; only then is the plan executed — one change at a time, verified before the next, with discrete commits.
  • Use it for "collaboration documents" too — audits and architecture explorations, not just plans. Repeatedly asking "explain this more, draw this out, how does this break, how do we recover" is how a one-pass shallow answer becomes the depth this report carries.

Continuity payoff: a plan that has been read and questioned before execution is the difference between "the agent changed the data layer and hopefully it's fine" and "we agreed what correct means, proved it with a failing test, and watched it go green."

4. Independent second-model review before trusting a fix

Christie connected the recurring-wipe problem to "having two models talk to each other," and that instinct is correct. For any change that touches data safety, get a second, independent model to review the plan and the implementation before the change is considered done — Codex, a Claude review partner, or OMP, whichever is available. The reviewer's job is to try to refute: find the unverified claim, the save path that was missed, the test that only checks a mock. This is the most direct defense against "said it fixed it, wasn't fixed," because the model that wrote the fix is the worst judge of whether it worked.

7. Operating habits that keep agents thorough

  • Stay on high or max effort. Lower thinking settings produce materially worse results; older sessions may have defaulted lower, so re-running important audits at high effort is worthwhile.
  • Use deep multi-agent audits for thoroughness. For "audit this very thoroughly" work, the heaviest setting (which fans out subagents across the codebase) is far more complete than a single pass — a good use of end-of-week tokens that would otherwise go unused.
  • Narrow asks go deep; wide asks stay shallow. Scope one row, one change, one write path at a time, verify it, commit it, then move on. The wider the body of work requested, the shallower the agent's coverage.
  • Sweep regularly. The unguarded write paths in this report are found by deep, targeted audits, not one-pass skims — make periodic targeted sweeps a habit.

15. Glossary of terms

This report uses a number of acronyms and technical terms. Each is defined twice below: a precise technical meaning for an engineer, and a plain in-plain-terms description that avoids jargon. The terms are grouped by topic.

Backup and recovery

TermTechnical meaningIn plain terms
Backup / snapshot / exportA point-in-time copy of data written somewhere it can be restored from later; an "export" writes data out to files or another storage location.A saved copy of the data as it was at a certain moment, kept so you can put it back if the live data is lost or damaged.
PITR — Point-in-Time RecoveryA Firestore feature that can restore the database to its exact state at any chosen moment within a recent window (currently up to 7 days).A rewind button for the database. If something is deleted or broken, you can return to how it looked a few minutes or days ago — but only within the last several days.
Scheduled backupsFirestore's managed feature that automatically saves backup copies on a set cadence (for example daily) with a chosen retention period.Automatic copies of the database saved on a regular timetable, so a recent saved version always exists without anyone remembering to make one.
Restore / restore drillLoading a backup into a database to recover data; a "drill" is a rehearsed restore into a non-production target to prove recovery works.Putting a saved copy back into use. A drill is a practice run that proves you can actually recover — instead of only assuming you can until a real emergency.
RunbookA documented, step-by-step operational procedure for a specific task, such as restoring a collection after data loss.A written checklist someone can follow under pressure, so recovery does not depend on one person remembering all the steps.
RPO — Recovery Point ObjectiveThe maximum acceptable amount of data, measured in time, that may be lost in an incident; it sets how frequently backups must run.How much recent work you can afford to lose. An RPO of one hour means backups run often enough that, at worst, about an hour of changes is lost.
RTO — Recovery Time ObjectiveThe maximum acceptable time to restore data or service after an incident; it shapes how the restore process is designed.How long an app can be down after a problem before it must be working again.

How data is saved — and lost

TermTechnical meaningIn plain terms
merge: true (Firestore set with merge)A write option that updates only the supplied fields and leaves unsupplied fields untouched — but still overwrites supplied fields, including ones sent as empty strings.A save setting that only changes the fields you send. The catch behind Classbook's data loss: if a field is sent blank, it still replaces the real value with blank.
Batch / bulk writeWriting many documents in a single operation.Saving or changing many records at once. Efficient, but one mistake can affect a large amount of data at the same time.
Awaited write vs. fire-and-forgetAn awaited write waits for the database to confirm the change persisted; a fire-and-forget write returns before confirmation, so failures go unnoticed."Awaited" means the app waits to hear that the save actually succeeded. "Fire-and-forget" means it assumes success and moves on — so a failed save can look fine while data is quietly lost.
Read-back verificationAfter writing, re-reading the data to confirm it was stored as intended.After saving, checking that the data really landed — catching a failed save immediately instead of days later.
Hard deletePermanently removing a record with no retained, recoverable copy.A permanent delete with no trash can. Once it is gone, it is gone.
Soft deleteMarking data as deleted but keeping it recoverable for a retention window instead of removing it immediately.A trash can. Deleted items are held for a while so they can be brought back, rather than vanishing instantly.
Object versioningKeeping previous versions of a file in Cloud Storage when it is overwritten or deleted.Automatically keeping older copies of a file, so overwriting or deleting it does not lose the previous version.
Lifecycle / retention rulesPolicies that automatically delete, archive, or move objects and old versions after set conditions.Automatic housekeeping rules — for example "keep deleted files for 30 days, then remove them" — to control cost and how long things are kept.
